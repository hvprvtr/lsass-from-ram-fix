# lsass-dump-toolkit

**An English version of this document is available in [`README-EN.md`](README-EN.md).**

Набор утилит для диагностики и починки дампов `lsass.exe`, снятых из **полного
образа оперативной памяти** или иными способами, не подгружающими вытесненные
страницы. Такие дампы часто не обрабатываются `pypykatz` / `mimikatz` — инструмент
падает с `LSA signature not found! / All detection methods failed`.

### Состав

| Скрипт | Назначение |
|--------|------------|
| **`fix_lsass_minidump.py`** | Диагностика (`--check`) и починка **готового minidump** (MDMP), вырезанного, например, через [MemProcFS](https://github.com/ufrisk/MemProcFS). Восстанавливает занулённые file-backed страницы модулей из оригинальных DLL. |
| **`binmap_to_minidump.py`** | Сборка настоящего minidump из **сырого дампа `bin`+`map`** (регионы памяти + текстовая карта `va length file_offset`). По пути выполняет такой же бэкфилл кода. |

Оба скрипта решают одну и ту же первопричину (см. ниже), просто на разных
входных форматах; логика бэкфилла кода у них общая.

---

## TL;DR

```bash
# 1. Диагностика: что повреждено и какие DLL нужно достать с целевой машины.
#    Ничего не пишет, оригинал не трогает. Файл DLL не требуется.
python3 fix_lsass_minidump.py dump.dmp --check

# 2. Починка: накладывает байты оригинальных DLL на зануленные страницы.
#    Пишет в НОВЫЙ файл, исходник не меняется.
python3 fix_lsass_minidump.py dump.dmp fixed.dmp \
    --module lsasrv.dll:/path/to/lsasrv.dll

# 3. Профит:
pypykatz lsa minidump fixed.dmp
```

А если на входе сырой дамп `bin`+`map` (не minidump) — сразу собрать и починить:

```bash
python3 binmap_to_minidump.py dump.bin out.dmp \
    --build 26200 --module lsasrv.dll:/path/to/lsasrv.dll
pypykatz lsa minidump out.dmp
```

## Проблема

### Симптом

Дамп `lsass`, снятый штатным способом (Task Manager → *Create dump file*,
`comsvcs.dll MiniDump`, procdump и т.п.), разбирается `pypykatz` без проблем.
А минидамп **той же машины**, полученный из образа физической RAM через
MemProcFS, — нет:

```
Exception: LSA signature not found!
Exception: All detection methods failed.
```

### Причина

Каждый процесс Windows видит **виртуальное** адресное пространство, но в
физической RAM в каждый момент лежит только его «рабочий набор» (working set).
Когда памяти не хватает, страницы вытесняются — и тип страницы определяет, куда:

* **Приватные/«грязные» страницы** (heap, стек, изменённые данные) — единственная
  копия их содержимого в памяти, поэтому при вытеснении ОС **записывает их в
  `pagefile.sys`**.
* **«Чистые» file-backed страницы** (код `.text` и read-only `.rdata`
  загруженных DLL) — это просто спроецированный с диска образ. При вытеснении ОС
  их **просто выбрасывает**, ничего не записывая: точная копия уже лежит в
  файле DLL на диске (`C:\Windows\System32\<dll>`). Понадобятся — будут
  перечитаны оттуда.

> Ключевой момент: выброшенная страница кода DLL **не попадает ни в RAM, ни в
> pagefile** — она остаётся только в файле DLL на диске.

**Почему дамп из Task Manager полный.** `MiniDumpWriteDump` выполняется в живой
ОС и читает виртуальную память процесса. Каждое чтение вытесненной страницы
вызывает page fault, и ОС прозрачно подгружает её обратно (приватную — из
pagefile, код — с диска). Дамп получается связным и полным.

**Почему дамп из образа RAM — нет.** winpmem/FTK/DumpIt снимают пассивный слепок
физической памяти: в нём есть только то, что **физически лежало в RAM** в момент
захвата. MemProcFS реконструирует виртуальное пространство `lsass` из этого
слепка; страницы, которые были вытеснены (чистый код DLL, или ушедшее в
pagefile/сжатый Store), взять неоткуда — и MemProcFS отдаёт их **нулями**.

Сигнатура LSA, по которой `pypykatz` находит crypto-ключи, лежит в коде
`lsasrv.dll` (`.text`). Если её страница вытеснена → занулена → сигнатура не
находится → разбор падает.

### Почему именно Windows 11

Это **не** категориальное отличие Win10/Win11 — механизм один и тот же. Вопрос
вероятностный: оказалась ли страница с сигнатурой резидентной в RAM в момент
захвата. Win11 (особенно 24H2/25H2, билды 26100/26200) за счёт более агрессивной
memory compression и подрезки рабочих наборов чаще вытесняет «холодный» код
простаивающего `lsass`, поэтому в RAM-образах Win11 нужные страницы отсутствуют
заметно чаще. На исход также влияют **условия захвата** (снят сразу после логона
под нагрузкой vs на «отстоявшейся» машине) и **сам инструмент** захвата.

---

## Решение

Потерянные страницы `.text`/`.rdata` — это неизменный, file-backed код. Его
можно **побайтово восстановить** из той же версии DLL, взятой с целевой машины:
в x64 код RIP-relative, релокации `.text` не затрагивают, поэтому код в памяти
идентичен коду на диске (проверено: все 323 страницы `.text` lsasrv.dll
совпали с эталонным Task-Manager-дампом, 0 различий).

Скрипт накладывает байты дисковой DLL на **только полностью нулевые** страницы
соответствующего модуля в минидампе. Реальные данные (`.data` с ключами, heap)
не трогаются.

После починки `pypykatz` находит сигнатуру и извлекает секреты:

```
== MSV ==
    Username: user
    NT: 57d583aa46d571502aad4bb7aea09c70   # совпадает с эталонным дампом
== DPAPI ==
    masterkey ...
```

## Использование

### Диагностика (`--check`)

Только считает зануленные страницы, **ничего не пишет**, **файл DLL не нужен**
(таблица секций читается из PE-заголовка прямо в дампе).

```bash
# Авто-скан всех известных credential-модулей:
python3 fix_lsass_minidump.py dump.dmp --check

# Конкретные модули:
python3 fix_lsass_minidump.py dump.dmp --check \
    --module lsasrv.dll --module kerberos.dll
```

Код возврата: `1` — найдены зануленные страницы, `0` — модули целы (удобно для
скриптов и пакетной проверки).

Пример вывода:

```
DIAGNOSE: dump.dmp
  [lsasrv.dll] base=0x... size=0x1b7000 : zero=156 nonzero=283  -> ПРОБЛЕМА
        .text      zero= 115 nonzero= 208  <-- дыры в коде
        .rdata     zero=  28 nonzero=  52  <-- дыры в коде
        .data      zero=   0 nonzero=  10
  [kerberos.dll] base=0x... : zero=335 nonzero=24 -> ПРОБЛЕМА  [PE-заголовок занулен, секции не разобрать]
  ...
ДОСТАТЬ С МАШИНЫ (file-backed код повреждён, нужен оригинал DLL):
   - lsasrv.dll
   - kerberos.dll
   ...
Затем чинить:  python3 fix_lsass_minidump.py <in> <out> --module lsasrv.dll:/путь/lsasrv.dll ...
```

Скрипт сам печатает готовую к вставке команду починки с нужными модулями.

### Починка

```bash
python3 fix_lsass_minidump.py <in.dmp> <out.dmp> \
    --module lsasrv.dll:/path/to/lsasrv.dll \
    [--module kerberos.dll:/path/to/kerberos.dll] \
    [--module msv1_0.dll:/path/to/msv1_0.dll] ...
```

* `<in.dmp>` — исходный минидамп (открывается **только на чтение**).
* `<out.dmp>` — выходной файл (создаётся как копия исходника, патчится копия).
* `--module NAME:PATH` — модуль и путь к оригинальной DLL **той же версии** с
  целевой машины. Можно повторять.

### Где взять оригинальные DLL

Файлы лежат на целевой машине в `C:\Windows\System32\`. **Версия должна точно
совпадать** с дампом. Если версия не та — скрипт всё равно отработает, но
наложит чужой код, и разбор будет некорректным. Соответствие можно
косвенно проверить: после починки `--check` целевого модуля должен дать `OK`, а
`pypykatz` — осмысленный результат.

## Известные модули (авто-скан)

`--check` без `--module` сканирует модули, в которых `pypykatz`/`mimikatz` ищут
сигнатуры и читают секреты:

| DLL | Что несёт |
|-----|-----------|
| `lsasrv.dll`  | MSV/NTLM, LSA-ключи, DPAPI, credman |
| `wdigest.dll` | WDigest (plaintext, если включён) |
| `kerberos.dll`| Kerberos (билеты, ключи) |
| `msv1_0.dll`  | MSV / SSP |
| `tspkg.dll`   | TsPkg (RDP) |
| `livessp.dll` | LiveSSP |
| `cloudap.dll` | CloudAP (Azure AD / PRT) |
| `dpapisrv.dll`| DPAPI |

## Сырые дампы `bin`+`map`: `binmap_to_minidump.py`

Некоторые дамперы пишут процесс не в формате minidump, а сырым способом:

* `<name>.bin` — голая конкатенация байт регионов памяти (без заголовков);
* `<name>.bin.map` — текстовая карта строк `va length file_offset`.

`pypykatz`/`mimikatz` такой формат не понимают. `binmap_to_minidump.py` собирает
из этой пары настоящий minidump (MDMP), синтезируя минимально необходимые потоки:

* **SystemInfoStream** — архитектура и `BuildNumber` (нужен для выбора шаблона
  структур; задаётся флагом `--build`);
* **ModuleListStream** — список модулей (нужен, чтобы найти `lsasrv.dll` и др.).
  Имя берётся из таблицы экспорта, а если она занулена/обрезана — по точному
  совпадению `SizeOfImage` с DLL, переданными через `--module`;
* **Memory64ListStream** — карта памяти (`va` → данные), построенная из `.map`.

Зачем нужен `.map`: в `.bin` нет ни заголовков, ни адресов — это просто склейка
кусков памяти. Карта — таблица трансляции «виртуальный адрес ↔ смещение в файле»,
прямой аналог потока `Memory64List` внутри minidump. Без неё байты `.bin`
бессмысленны для адресного разбора.

По ходу сборки скрипт выполняет **тот же бэкфилл кода**, что и
`fix_lsass_minidump.py`: занулённые file-backed страницы указанных модулей
заполняются байтами оригинальных DLL.

### Использование

```bash
python3 binmap_to_minidump.py dump.bin out.dmp \
    --build 26200 \
    --module lsasrv.dll:/path/to/lsasrv.dll \
    [--module kerberos.dll:/path/to/kerberos.dll] ...

pypykatz lsa minidump out.dmp
```

* `dump.bin` — сырой дамп; карта ищется как `dump.bin.map` либо `dump.map`.
* `--build` — `BuildNumber` целевой ОС (например, `26200` для Win11 25H2).
* `--module NAME:PATH` — DLL для бэкфилла кода и для надёжного именования модуля
  по `SizeOfImage`. Повторяемо. Для извлечения NTLM достаточно `lsasrv.dll`;
  добавь `kerberos.dll`/`wdigest.dll`/`dpapisrv.dll` и т.д. для остальных
  провайдеров.

Файл `.bin` целиком копируется в выходной minidump как блок памяти — исходные
`bin`/`map` не изменяются.

## Ограничения

* **Heap не восстанавливается.** Скрипт чинит только file-backed код DLL. Если
  из RAM были вытеснены **приватные heap-страницы** `lsass` (где лежат сами
  структуры MSV/Kerberos и ключевой материал), их единственная копия была в
  RAM/pagefile — в чистом RAM-образе их нет, восстановить из DLL нельзя. На
  практике это выглядит так: сигнатура и часть секретов (например, DPAPI)
  извлекаются, но обход списка логон-сессий падает на «битом» указателе из
  зануленной heap-страницы. Вывод: бэкфилл DLL — **необходимое**, но не всегда
  **достаточное** условие.
* **Нужна точная версия DLL.** См. выше.
* **Патч по правилу «страница целиком нулевая».** Скрипт не отличает реально
  отсутствующую страницу от легитимно-нулевой. Для кода (`.text`/`.rdata`) это
  безопасно (нулевых страниц там при нормальной работе нет, а байты с диска
  совпадают). Теоретический пограничный случай — нулевая страница в `.data`
  (zero-initialized данные): скрипт перезапишет её образом с диска. На практике
  `.data` с ключами всегда резидентен и не затрагивается.
* **Конвертер не добавляет отсутствующих данных.** `binmap_to_minidump.py`
  работает только с тем, что есть в `bin`+`map`. Если дампер сохранил неполный
  набор регионов — например, обрезал образы модулей и в дамп не попала секция
  `.data` lsasrv.dll (где лежат ключи) или heap, — конвертация и бэкфилл кода не
  помогут: `.data`/heap не file-backed, восстановить их неоткуда. Проверять
  полноту стоит до конвертации: секции `.text` **и** `.data` lsasrv.dll должны
  присутствовать в карте.

### Практическая рекомендация

Для гарантированного извлечения всех типов кредов (особенно Kerberos) лучше
снимать образ RAM при «горячем» `lsass` — сразу после/во время аутентификации,
когда его рабочий набор резидентен. Разные инструменты захвата дают разную
полноту даже на одной машине.

## Как это работает (кратко)

**`fix_lsass_minidump.py`** (готовый minidump):

1. Парсит минидамп пакетом `minidump`, берёт список модулей (база, размер) и
   таблицу memory-сегментов (виртуальный адрес ↔ смещение в файле дампа).
2. Для целевого модуля идёт по его страницам; если страница в дампе **целиком
   нулевая** — вычисляет соответствующий RVA → файловое смещение в дисковой DLL
   (по таблице секций PE) и пишет диск-байты в копию дампа.
3. В режиме `--check` ничего не пишет, только классифицирует страницы по секциям
   и формирует список модулей, которые нужно достать с машины.

**`binmap_to_minidump.py`** (сырой `bin`+`map`):

1. Читает карту (`va length file_offset`) и сканирует регионы на PE-заголовки,
   определяя модули (база, `SizeOfImage`, имя из экспорта/по `--module`).
2. Бэкфиллит занулённые file-backed страницы модулей из дисковых DLL — прямо в
   блоке памяти будущего minidump (по той же логике, что и `fix_lsass_minidump.py`).
3. Записывает minidump: заголовок MDMP, потоки `SystemInfo`/`ModuleList`/
   `Memory64List` и сам блок памяти (содержимое `.bin` в порядке карты).
